## HIGH (5) — Active risk

### 1. Stored XSS in scoreboard rendering

**File:line** `client/src/ui/HUD.ts:1024`, `NetworkClient.ts:962`

**Issue** — `buildScoreboard()` interpolates peer-controlled `playerName` into `innerHTML` with no HTML escaping. Server validates only non-empty, length ≤20, unique. Attacker payload like `<svg/onload=...>` (fits 20 chars) broadcasts via roster, replayed, and rendered as live HTML in every opponent's `GAME_OVER`/`ROUND_OVER` scoreboard.

**Severity** — HIGH

**Fix** — Build scoreboard rows with `createElement()` + `.textContent` (safe) or apply HTML-escape utility to `playerName` before interpolation. Optionally strip `<>` in server name validators.

### 10. Auth — Replay attack risk on submit_action

**Issue** — `submit_action` RPC validates turn ownership via action log (seq-based), no timestamp nonce. If Realtime broadcast is replayed in-order, replay is prevented by seq uniqueness. If attacker captures raw HTTP POST, replay would be rejected by seq-already-exists. Design sound but worth explicit threat model documentation.

**Scope** — Auth/crypto; design confirmed safe.

### 11. Migration — Supabase Realtime subscription model scalability

**Issue** — Current design: every room broadcasts all actions via Realtime; every client listens to one room's channel. Scales to ~dozens of concurrent rooms on t3.micro. At scale (hundreds of concurrent rooms) or seriousness gate, backend strategy may shift to Durable Objects or PartyKit (per CONFIRM-03). No action needed now; flagged as strategic decision point.

**Scope** — Migration / backend; documented in CONTEXT.md strategic direction.
